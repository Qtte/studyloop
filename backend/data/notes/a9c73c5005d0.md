# Redis 基础模型、数据类型与为什么用它做缓存

## 先记住最核心的模型

Redis 顶层可以先理解成一个全局的 `key -> value` 哈希表：

```text
key -> value
```

其中：

- `key` 是用户自己定义的名字
- `value` 是真正存储的数据对象

例如：

```text
"name"   -> String
"orders" -> List
"tags"   -> Set
"user:1" -> Hash
```

## Redis 里的 key 和 value 怎么理解

### key

由用户自己定义，用来定位数据，比如：

```text
SET name "Alice"
LPUSH orders order1 order2
SADD tags vip active
```

这里的 `name`、`orders`、`tags` 都是 key。

### value

value 不是只有字符串，它支持多种数据结构，常见有：

- `String`
- `List`
- `Set`
- `Hash`
- `ZSet`

所以 Redis 不是“只能存字符串的缓存”，而是“顶层是 key-value，但 value 可以是不同类型的结构化对象”。

## 查询一个 key 时大致发生了什么

可以先用一个工程化心智模型理解：

1. 对用户传入的 key 计算 hash
2. 根据 hash 定位到哈希表中的某个桶
3. 在桶内继续比较 key
4. 找到匹配项后返回对应 value

更稳的表述是：

> Redis 顶层用字典结构管理 key，查找时会先通过 hash 快速定位，再处理哈希冲突并比对实际 key。

## Redis 和 Memcached 的区别

### 相同点

- 都基于内存
- 都常被当作缓存
- 都支持过期时间
- 性能都很高

### Redis 更常被选中的原因

- Redis 支持更多数据类型
- Redis 支持持久化
- Redis 原生支持更丰富的功能
- Redis 有更完整的生态和集群能力

### 一个简化对比

| 维度 | Redis | Memcached |
| --- | --- | --- |
| 数据类型 | 丰富 | 更简单，偏纯 KV |
| 持久化 | 支持 | 不支持 |
| 集群能力 | 原生支持较完整 | 更多依赖客户端 |
| 功能扩展 | Lua、发布订阅、事务等 | 相对少 |

所以如果只是非常纯粹的简单缓存，Memcached 也能做；但大多数业务最终还是更偏向 Redis，因为它的能力边界更宽。

## 为什么 Redis 很适合做 MySQL 缓存

主要是两点：

### 1. 高性能

- MySQL 很多读操作最终会落到磁盘或更重的存储链路上
- Redis 主要操作内存，访问速度更快

### 2. 高并发

- Redis 单机 QPS 通常明显高于 MySQL
- 把热点读请求挡在 Redis，可以大幅减轻数据库压力

一句话理解：

> Redis 适合放在 MySQL 前面承接高频读请求，让一部分请求直接在缓存层结束，而不是每次都打到数据库。

## 但 Redis 不是万能数据库替身

把 Redis 当缓存时，要记住两个边界：

- 它适合热点、读多写少、可容忍短时最终一致的数据
- 不适合轻率地当订单状态、支付状态这类强一致数据的唯一事实来源

## 一句话总结

Redis 顶层是 `key -> value` 的内存型字典结构，但 value 支持多种数据类型；它之所以常被放到 MySQL 前面做缓存，本质上是因为它更快、更能扛并发。
